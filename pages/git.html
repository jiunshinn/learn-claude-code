🔀 Git

Git 완전 정복

파일의 변경 이력을 관리하는 Git. 실습과 함께 배우면 전혀 어렵지 않아요.

📓 비유로 이해하기

"보고서_최종.docx", "보고서_최종_진짜최종.docx", "보고서_최종_수정_v3.docx" — 이렇게 파일을 복사해본 적 있나요?

Git은 이런 버전 관리를 자동으로 해주는 도구예요. 파일을 하나만 유지하면서도, 언제 누가 무엇을 바꿨는지 전부 기록해 줍니다. 실수로 파일을 망쳐도 이전 버전으로 쉽게 되돌릴 수 있어요.

구글 문서의 "버전 기록" 기능을 상상해 보세요. Git은 그것의 훨씬 강력한 버전이에요 — 모든 종류의 파일에 대해, 내 컴퓨터에서 동작합니다.

📎 잠깐! Git은 코드만 관리하는 도구가 아니에요

"Git = 개발자 도구"라고 생각하기 쉽지만, 실제로는 모든 종류의 파일을 관리할 수 있어요.

  • 기획 문서 — 기획서, 제안서, 회의록의 변경 이력 관리
  • 디자인 파일 — Figma 연동이 아니라 SVG, HTML 기반 디자인 파일
  • 데이터 파일 — CSV, JSON 등 데이터 파일의 버전 추적
  • 웹사이트 — HTML/CSS 파일을 GitHub Pages로 무료 배포
  • 글쓰기 — 블로그 글, 번역 원고, 매뉴얼 등 텍스트 기반 작업

즉, "파일이 바뀌는 모든 작업"에 Git을 쓸 수 있어요. 코딩을 안 해도 Git은 여러분의 강력한 도구가 될 수 있습니다.

🧠 AI 시대, 명령어를 외울 필요 없어요

과거에는 Git을 쓰려면 수십 개의 명령어를 외워야 했어요. 하지만 이제는 다릅니다.

Claude Code + gh CLI 조합이면, 여러분은 개념만 알면 충분해요. 아래 6가지 개념만 이해하면 나머지는 AI가 해줍니다:

📁 저장소 (Repository) 🌿 브랜치 (Branch) 💾 커밋 (Commit) 🚚 푸시 (Push) 📬 풀 리퀘스트 (PR) 📥 풀 (Pull)

예를 들어 "feature 브랜치에서 작업한 걸 PR 만들어줘"라고 말하면, Claude Code가 git push하고 gh pr create까지 알아서 해줘요. 명령어를 몰라도 "무엇을 하고 싶은지"만 말하면 돼요.

🤔 비전공자가 Git을 배워야 하는 이유

코딩을 안 해도 Git이 필요한 순간이 점점 많아지고 있어요.

🤖

Claude Code의 안전장치

Claude Code가 파일을 수정할 때, Git이 있으면 언제든 원래 상태로 되돌릴 수 있어요. AI가 실수해도 걱정 없어요.

👥

팀 협업의 기본

개발자와 함께 일하거나, 문서/디자인 파일을 공유할 때 Git은 팀 협업의 공용어예요.

📂

GitHub 포트폴리오

GitHub에 프로젝트를 올려두면 나만의 디지털 포트폴리오가 돼요. 웹사이트도 무료로 배포할 수 있어요.

⏰

시간 여행

"3일 전 버전으로 돌아가고 싶어" — Git이 있으면 시간 여행이 가능해요. 실수가 두렵지 않아요.

🧩 핵심 개념 6가지

이것만 이해하면 돼요. 명령어는 AI가 대신 쳐줍니다.

📁

저장소 (Repository)

Git이 변경 이력을 추적하는 프로젝트 폴더예요. git init 명령으로 만들거나, GitHub에서 가져올 수 있어요.

  • 프로젝트의 모든 파일과 변경 기록이 여기에
  • 로컬(내 컴퓨터)과 원격(GitHub)이 있음
💾

커밋 (Commit)

"이 시점을 기억해줘"라고 저장하는 거예요. 게임의 세이브 포인트와 같아요.

  • 변경 내용 + 설명 메시지를 함께 기록
  • 언제든 이 시점으로 되돌아갈 수 있음
🌿

브랜치 (Branch)

원본을 건드리지 않고 새로운 작업을 시도하는 "복사본 공간"이에요.

  • 실험적인 변경을 안전하게 시도 가능
  • 완성되면 원본에 합칠 수 있음 (Merge)
☁️

푸시 / 풀 (Push / Pull)

내 컴퓨터의 변경 사항을 GitHub에 올리거나(Push), GitHub의 변경 사항을 가져오는(Pull) 거예요.

  • Push: 내 작업을 팀과 공유
  • Pull: 다른 사람의 변경 사항을 받아오기
📬

풀 리퀘스트 (Pull Request / PR)

"내가 이렇게 바꿔봤는데, 확인하고 합쳐줄래?" 하고 팀에게 검토를 요청하는 거예요.

  • 브랜치에서 작업 → PR 생성 → 리뷰 → 합치기
  • 혼자 작업할 때도 변경 내역을 정리하는 데 유용
  • GitHub에서 변경 사항을 한눈에 비교 가능
🔀

머지 (Merge)

브랜치에서 작업한 내용을 원본(main)에 합치는 것이에요. PR이 승인되면 머지 버튼 하나로 끝!

  • PR 리뷰가 끝나면 "Merge" 클릭
  • 충돌이 있으면 Claude Code가 해결 도와줌

🛠️ 실습 1: Git 설치 확인 & 초기 설정

터미널을 열고 하나씩 따라해 보세요. 5분이면 끝나요!

1

Git이 설치되어 있는지 확인하기

터미널을 열고 아래 명령어를 입력하세요. 버전 번호가 나오면 이미 설치된 거예요!

git --version
💡 Mac에는 Git이 기본 설치되어 있어요. Windows는 git-scm.com에서 다운로드하세요. 설치 옵션은 모두 기본값으로 "Next"만 누르면 돼요.
2

내 이름과 이메일 등록하기

Git에게 "나는 누구"인지 알려주세요. 커밋할 때 이 정보가 기록돼요. 처음 한 번만 하면 돼요.

git config --global user.name "홍길동"
git config --global user.email "gildong@email.com"
💡 GitHub 계정이 있다면, GitHub에 등록한 이메일과 같은 이메일을 사용하세요.
3

설정이 잘 됐는지 확인하기

아래 명령어로 방금 설정한 정보를 확인할 수 있어요.

git config --list

📂 실습 2: 첫 번째 Git 저장소 만들기

연습용 폴더를 만들고, Git으로 변경 이력을 추적해 봐요.

1

연습용 폴더 만들기

바탕화면(Desktop)에 연습용 폴더를 만들고 이동할게요.

Mac

mkdir ~/Desktop/git-practice && cd ~/Desktop/git-practice

Windows

mkdir %USERPROFILE%\Desktop\git-practice && cd %USERPROFILE%\Desktop\git-practice
2

Git 저장소 초기화하기

이 폴더를 Git이 관리하는 저장소로 만들어요.

git init

✅ "Initialized empty Git repository..." 메시지가 나오면 성공! 이제 이 폴더의 모든 변경이 추적돼요.

3

첫 번째 파일 만들기

간단한 텍스트 파일을 하나 만들어 볼게요.

Mac

echo "안녕하세요! Git 연습 중입니다." > hello.txt

Windows

echo 안녕하세요! Git 연습 중입니다. > hello.txt
4

현재 상태 확인하기

git status는 "지금 뭐가 바뀌었어?"라고 Git에게 물어보는 거예요.

git status

📋 hello.txt가 빨간색으로 "Untracked files"에 나타나요. "아직 추적하지 않는 새 파일이 있어요"라는 뜻이에요.

5

파일을 박스에 담기 (Staging)

커밋할 파일을 선택하는 과정이에요. 택배 비유로 치면 "박스에 물건 담기"예요.

git add hello.txt
💡 git add .을 쓰면 변경된 모든 파일을 한꺼번에 담을 수 있어요. 하지만 처음에는 파일 이름을 직접 지정하는 게 안전해요.
6

첫 번째 커밋! (세이브 포인트 만들기)

드디어 첫 번째 저장이에요! -m 뒤에 "무엇을 했는지" 메시지를 적어요.

git commit -m "첫 번째 커밋: hello.txt 파일 추가"

🎉 축하해요! 방금 첫 번째 Git 커밋을 만들었어요!

7

커밋 기록 확인하기

지금까지의 기록을 볼 수 있어요. 게임의 세이브 목록처럼요.

git log --oneline

📋 방금 만든 커밋이 해시코드(예: a1b2c3d)와 함께 표시돼요.

🔁 실습 3: 파일 수정하고 다시 커밋하기

Git의 핵심은 "수정 → add → commit"의 반복이에요. 한 번 더 해볼까요?

1

파일 내용 수정하기

hello.txt에 한 줄을 추가해 볼게요.

Mac

echo "Git 정말 재미있네요!" >> hello.txt

Windows

echo Git 정말 재미있네요! >> hello.txt
💡 > 하나는 파일을 덮어쓰고, >> 두 개는 기존 내용 뒤에 추가해요. 주의하세요!
2

무엇이 변경됐는지 확인하기

git diff는 "마지막 커밋과 뭐가 달라졌어?"를 보여줘요.

git diff

초록색 +로 표시된 줄이 새로 추가된 내용이에요.

3

add → commit 한 번에!

이번에는 add와 commit을 한 줄로 해볼게요.

git add hello.txt && git commit -m "hello.txt에 감상 한 줄 추가"
4

커밋 기록 다시 확인!

이제 커밋이 2개가 됐을 거예요.

git log --oneline

🎯 두 개의 커밋이 시간순으로 나타나요. 이게 바로 버전 관리예요!

⏪ 실습 4: 실수를 되돌리기

Git의 가장 강력한 기능! 실수해도 이전 상태로 돌아갈 수 있어요.

1

일부러 실수해보기

실수를 시뮬레이션해볼게요. 파일 내용을 엉뚱하게 바꿔봐요.

Mac

echo "아이고 실수!" > hello.txt

Windows

echo 아이고 실수! > hello.txt

😱 이제 원래 내용이 사라지고 "아이고 실수!"만 남았어요.

2

마지막 커밋 상태로 되돌리기

걱정 마세요! 아직 커밋하지 않았으니, 마지막으로 저장한 상태로 되돌릴 수 있어요.

git checkout -- hello.txt

✅ 파일을 열어보면 원래 내용이 복구되어 있을 거예요!

💡 git checkout -- 파일명은 "이 파일을 마지막 커밋 상태로 되돌려줘"라는 뜻이에요. 아직 커밋하지 않은 변경사항만 되돌릴 수 있어요.
!

커밋한 후에 되돌리고 싶다면?

이미 커밋한 내용을 되돌리고 싶을 때는 상황에 따라 다른 명령어를 써요. 하지만 걱정하지 마세요 — Claude Code에게 말로 설명하면 알아서 해줘요!

마지막 커밋을 취소하고 이전 상태로 돌아가줘

☁️ 실습 5: GitHub에 내 프로젝트 올리기

로컬에서 만든 프로젝트를 GitHub에 올려서 백업하고 공유해 봐요.

1

GitHub 계정 만들기

github.com에서 무료 계정을 만드세요. 이메일만 있으면 돼요.

2

새 저장소(Repository) 만들기

GitHub 우측 상단의 "+" 버튼 → "New repository"를 클릭하세요.

  • Repository name: git-practice 입력
  • Public 선택 (누구나 볼 수 있음)
  • "Add a README file"은 체크하지 마세요!
  • "Create repository" 클릭
3

로컬과 GitHub 연결하기

GitHub에서 만든 저장소의 주소를 연결해요. GitHub 페이지에 나온 명령어를 복사해서 터미널에 붙여넣으세요.

git remote add origin https://github.com/내아이디/git-practice.git
💡 내아이디 부분을 본인의 GitHub 아이디로 바꿔주세요!
4

GitHub에 푸시하기!

드디어 내 코드를 GitHub에 올리는 순간이에요!

git branch -M main && git push -u origin main

🎉 GitHub에서 내 저장소 페이지를 새로고침하면, 방금 올린 파일이 보일 거예요!

💡 처음 푸시할 때 GitHub 로그인을 요청할 수 있어요. 비밀번호 대신 Personal Access Token을 사용하세요. GitHub 설정(Settings) → Developer settings → Personal access tokens에서 만들 수 있어요.

🌿 실습 6: 브랜치로 안전하게 실험하기

원본을 보존하면서 새로운 시도를 해보는 브랜치를 직접 만들어 봐요.

1

새 브랜치 만들고 이동하기

"experiment"라는 이름의 브랜치를 만들고 그쪽으로 이동해요.

git checkout -b experiment

🌿 이제 "experiment" 브랜치에 있어요. 여기서 뭘 해도 원본(main)에는 영향이 없어요!

2

브랜치에서 새 파일 추가하기

실험적으로 새 파일을 만들어 볼게요.

Mac

echo "이건 실험용 파일이에요!" > experiment.txt

Windows

echo 이건 실험용 파일이에요! > experiment.txt
git add experiment.txt && git commit -m "실험용 파일 추가"
3

원본(main)으로 돌아가 보기

main 브랜치로 돌아가면 experiment.txt가 사라져요! 놀라지 마세요 — experiment 브랜치에만 있는 파일이니까요.

git checkout main

👀 폴더를 확인해보면 experiment.txt가 없어요. main에는 영향이 없는 거예요!

4

실험 결과 합치기 (Merge)

실험이 마음에 들면, main에 합칠 수 있어요.

git merge experiment

✅ 이제 main에도 experiment.txt가 생겼어요! 브랜치의 변경사항이 합쳐진 거예요.

📋 꼭 알아야 할 Git 명령어 요약

이 명령어들만 알면 일상적인 Git 사용에 충분해요.

명령어 쉬운 설명 비유
git init이 폴더를 Git으로 관리 시작일기장 시작
git status지금 뭐가 변경됐는지 확인거울 보기
git add 파일명커밋할 파일 선택박스에 담기
git commit -m "메시지"변경 사항 저장세이브 포인트
git log --oneline저장 이력 보기세이브 목록
git diff뭐가 달라졌는지 비교틀린 그림 찾기
git checkout -b 이름새 브랜치 만들고 이동연습장 꺼내기
git merge 브랜치명브랜치 합치기연습장 내용 본책에 옮기기
git pushGitHub에 업로드택배 발송
git pullGitHub에서 다운로드택배 수령

🤖 Claude Code가 Git을 쉽게 만들어줘요

명령어를 외울 필요 없어요. Claude에게 말로 시키면 돼요.

1

자동 커밋 메시지

2

자연어로 Git 조작

"새 브랜치 만들어줘", "마지막 변경 사항 되돌려줘" 같은 일상 표현으로 Git을 다룰 수 있어요.

3

충돌 해결 도우미

여러 사람이 같은 파일을 수정해서 충돌이 나면, Claude가 차이점을 분석하고 해결 방법을 알려줘요.

4

변경 이력 검색

"이 파일이 마지막으로 언제 수정됐어?", "이 부분을 왜 바꿨는지 알려줘" 같은 질문도 가능해요.

💬 이렇게 말해보세요

실제로 Claude Code에서 바로 사용할 수 있는 프롬프트예요.

🆕

Git 시작하기

새 프로젝트에 Git을 처음 설정할 때

프롬프트 이 폴더에 Git 저장소를 초기화하고, .gitignore 파일도 만들어줘. Node.js 프로젝트야.
💾

변경 사항 저장하기

작업한 내용을 커밋으로 기록할 때

프롬프트 지금까지 변경한 내용을 정리해서 커밋 메시지와 함께 저장해줘.
🔍

변경 이력 확인하기

어떤 파일이 언제, 왜 바뀌었는지 알고 싶을 때

프롬프트 최근 일주일간 이 프로젝트에서 변경된 파일 목록을 보여줘. 각 변경의 요약도 알려줘.
⏪

실수 되돌리기

잘못된 변경을 이전 상태로 복구할 때

프롬프트 방금 수정한 config.json 파일을 수정 전 상태로 되돌려줘.
🌿

브랜치로 안전하게 작업하기

원본을 유지하면서 새로운 기능을 시도할 때

프롬프트 "new-design"이라는 브랜치를 만들어줘. 여기서 스타일을 실험해볼 거야.
☁️

GitHub에 올리기

내 작업을 GitHub 원격 저장소에 공유할 때

프롬프트 지금까지 작업한 내용을 GitHub에 푸시해줘.

🚨 자주 하는 실수 & 해결법

초보자가 겪는 흔한 문제들, 당황하지 마세요!

!

"커밋할 게 없다"는 메시지가 나올 때

git add를 먼저 했는지 확인하세요. add 없이는 커밋할 파일이 없어요.

git add . && git commit -m "변경사항 저장"
!

push할 때 "rejected" 에러

다른 사람(또는 GitHub에서)이 먼저 변경한 게 있을 때 발생해요. 먼저 pull 하세요.

git pull origin main
!

잘못된 파일을 add 했을 때

아직 커밋하지 않았다면, staging에서 빼면 돼요. 파일 내용은 안 바뀌어요.

git reset HEAD 파일명
!

커밋 메시지를 잘못 썼을 때

가장 최근 커밋의 메시지만 바꿀 수 있어요.

git commit --amend -m "수정된 커밋 메시지"
💡 이미 push한 커밋은 amend하면 복잡해져요. 그럴 땐 Claude Code에게 도움을 요청하세요!

🙈 .gitignore — Git이 무시할 파일 정하기

비밀번호, 임시 파일 등 Git에 올리면 안 되는 것들을 지정해요.

📝 .gitignore 파일 예시

프로젝트 루트에 .gitignore 파일을 만들고, 무시할 패턴을 적어요.

# 비밀 정보가 담긴 파일 .env secrets.json # 자동 생성되는 폴더 node_modules/ __pycache__/ # OS가 만드는 불필요한 파일 .DS_Store Thumbs.db
💡 Claude Code에게 "이 프로젝트에 맞는 .gitignore 만들어줘"라고 하면 자동으로 만들어 줘요!

⚡ Claude Code + gh CLI = 비전공자의 무기

Claude Code와 GitHub CLI(gh)를 함께 쓰면, 개발자처럼 Git을 쓸 수 있어요. 명령어 하나 모르고도요.

🔧 gh CLI란?

gh는 GitHub의 공식 명령줄 도구예요. 터미널에서 PR 만들기, 이슈 확인, 저장소 만들기 등을 할 수 있어요. Claude Code가 gh 명령어를 자동으로 실행해 주니까, 여러분은 설치만 해두면 돼요.

# Mac brew install gh # Windows winget install GitHub.cli
# 설치 후 로그인 (처음 한 번만) gh auth login

🎯 실전 시나리오: 비전공자의 하루

1

아침 — 프로젝트 시작

Claude Code에게 말하기:

이 폴더를 Git으로 관리 시작하고, GitHub에 저장소도 만들어줘. "my-project"라는 이름으로.

→ Claude가 git init, gh repo create, 초기 커밋, 푸시를 알아서 해줘요.

2

오전 — 새 기능 작업

Claude Code에게 말하기:

"add-contact-page"라는 브랜치를 만들고, 거기서 연락처 페이지를 추가해줘.

→ Claude가 브랜치 생성, 파일 작업, 커밋까지 해줘요. 원본(main)은 안전!

3

오후 — PR 만들기

Claude Code에게 말하기:

지금까지 작업한 내용으로 PR을 만들어줘.

→ Claude가 git push 후 gh pr create로 PR을 자동 생성! 제목과 설명도 작성해 줘요.

4

퇴근 전 — PR 확인 & 머지

Claude Code에게 말하기:

열려있는 PR 목록 보여줘. 문제 없으면 머지해줘.

→ Claude가 gh pr list로 확인하고, gh pr merge로 합쳐줘요.

✨ 핵심 포인트: 위 시나리오에서 여러분이 직접 친 Git/gh 명령어는 0개예요. 개념을 이해하고 "무엇을 하고 싶은지"만 말하면, Claude Code가 올바른 명령어를 골라 실행해 줍니다. 비전공자에게 필요한 건 명령어 암기가 아니라 개념의 이해예요.

💡 비개발자를 위한 최종 팁

명령어를 외우는 시대는 끝났어요. Claude Code + gh CLI가 있으면, 여러분은 저장소, 브랜치, 커밋, 푸시, PR, 풀 — 이 6가지 개념만 이해하면 됩니다.

"브랜치 만들어줘", "커밋해줘", "PR 만들어줘"처럼 개념을 담은 한 마디면 Claude가 나머지를 해줘요. 위의 실습으로 개념을 체득한 뒤에는, 실전에서는 Claude에게 말하기만 하세요.

기억하세요: Git이 있으면 실수가 두렵지 않아요. 언제든 되돌릴 수 있으니까요. 그리고 Git은 코드만의 도구가 아니에요 — 문서, 데이터, 웹사이트, 어떤 파일이든 관리할 수 있습니다.
